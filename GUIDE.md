# Claude Code + Open Code 협업 전략 가이드

> 설계는 Claude Code, 구현은 Open Code, 검증은 GitHub.
> 세 역할은 **문서로만** 대화한다.

---

## 1. 왜 나누는가

한 에이전트에게 설계·구현·검증을 모두 맡기면 세 가지가 무너진다.

| 문제 | 설명 |
|---|---|
| **자기 검증 편향** | 자기가 짠 코드를 자기가 리뷰하면 통과시킬 이유부터 찾는다. |
| **컨텍스트 오염** | 설계 논의 수십 턴이 쌓인 상태에서 구현하면, 결정된 사항과 폐기된 아이디어가 뒤섞인다. |
| **비용 비효율** | 스펙이 확정된 뒤의 타이핑 작업에까지 최상위 모델을 쓸 이유가 없다. |

역할을 쪼개면 각 단계가 **다른 컨텍스트, 다른 모델, 다른 판단 기준**으로 돌아간다.
설계자가 놓친 걸 구현자가 질문하고, 둘 다 놓친 걸 CI가 잡는다.

---

## 2. 역할 계약

### 설계 — Claude Code

**한다**
- 코드베이스 탐색, 현재 동작 파악, 진입점 식별
- 문제를 "무엇을 왜"로 정의하고 요구사항으로 분해
- 수정 대상 파일과 제약 조건을 **경로:라인** 수준으로 명시
- 완료 기준(검증 가능한 형태)을 정의
- `prompts/` 에 핸드오프 문서 작성

**안 한다**
- 구현. 스펙을 쓰다가 "이건 그냥 내가 고치는 게 빠른데" 라는 생각이 들면, 그건 스펙이 충분히 좋다는 신호지 직접 고치라는 신호가 아니다.
- 애매한 채로 넘기기. 설계가 결정하지 않은 건 구현이 **추측**한다.

### 구현 — Open Code

**한다**
- 핸드오프 문서 하나만 읽고 그대로 구현
- 스펙과 실제 코드가 다르면 **멈추고 보고**
- 커밋, 브랜치, PR 생성

**안 한다**
- 범위 밖 리팩터링 (문서에 없으면 하지 않는다)
- 스펙 재해석. 모호하면 채우지 말고 되묻는다.
- 테스트를 통과시키려고 테스트를 고치기

### 검증 — GitHub

**한다**
- CI: 테스트 / 타입체크 / 린트 — 사람도 에이전트도 못 우기는 객관적 게이트
- PR diff: 스펙 대비 실제 변경 범위 확인
- 리뷰 코멘트: 다음 사이클의 입력

**핵심**: 검증은 **로컬이 아닌 곳**에 있어야 한다. 구현자가 자기 환경에서 "통과했다"고 말하는 것과 CI가 초록불을 켜는 건 다른 사건이다.

## 3. 핵심 원칙 — 핸드오프 문서가 유일한 인터페이스

```
Claude Code          prompts/xxx.md          Open Code           GitHub
  (설계)      ──────────>  스펙  ──────────>  (구현)  ──────────>  (검증)
                                                                    │
     └──────────────── PR 리뷰 / CI 실패 ◀───────────────────────────┘
```

Claude Code의 대화 로그, 사고 과정, 탐색 결과는 **Open Code에 전달되지 않는다**.
문서에 안 적힌 건 존재하지 않는 것이다.

이 제약이 불편해 보이지만, 실은 이 방식의 핵심 가치다.
**문서로 설명할 수 없는 설계는 아직 설계가 아니다.** 핸드오프 문서를 쓰는 행위 자체가 설계 검증이다.

> 판별법: 그 문서를 이 프로젝트를 처음 보는 사람에게 줬을 때 구현이 되는가?
> 안 되면 설계가 덜 된 것이다.

## 4. 워크플로우

> 실제 명령·프롬프트·체크리스트가 포함된 상세 절차는 [WORKFLOW.md](WORKFLOW.md) 참고.

### 1단계 — 설계 (Claude Code)

```
"로그인 실패 시 에러 메시지가 안 뜨는 문제. prompts/ 에 버그픽스 스펙 작성해줘"
```

Claude Code가 코드를 탐색하고 `prompts/2026-07-16-login-error-message.md` 를 만든다.
`_TEMPLATE-bugfix.md` 형식을 따른다.

**사람이 여기서 검토한다.** 스펙이 틀리면 구현도 틀린다. 이 단계의 5분이 뒤의 한 시간을 아낀다.

### 2단계 — 구현 (Open Code)

새 터미널, 새 컨텍스트에서:

```
"prompts/2026-07-16-login-error-message.md 를 읽고 그대로 구현해줘.
 문서에 없는 건 하지 말고, 스펙과 코드가 다르면 멈추고 알려줘."
```

### 3단계 — 검증 (GitHub)

PR을 올리고 CI를 돌린다. 결과에 따라:

| 상황 | 되돌아갈 곳 |
|---|---|
| CI 실패 — 구현이 스펙을 못 지킴 | **2단계** (Open Code 재작업) |
| CI 통과 — 근데 원하던 게 아님 | **1단계** (스펙이 틀렸다) |
| 스펙과 코드 현실이 다름 | **1단계** (설계 재탐색) |

**CI 통과했는데 원하던 게 아닌 경우를 구현자에게 되돌리지 말 것.** 그건 설계 실패다.
구현자를 붙잡고 고쳐나가면 스펙은 영영 틀린 채로 남고, 다음 작업도 같은 자리에서 틀린다.

## 5. 경계 — 어디에 무엇을 맡기나

| 작업 | 담당 | 이유 |
|---|---|---|
| "이 코드베이스가 어떻게 돌아가지?" | Claude Code | 탐색·종합 |
| "이 버그 원인이 뭐야?" | Claude Code | 가설 수립·검증 |
| "어떤 설계가 나아?" | Claude Code | 트레이드오프 판단 |
| "스펙대로 이 3개 파일 고쳐" | Open Code | 기계적 실행 |
| "이 패턴을 40군데 적용해" | Open Code | 반복 작업 |
| "테스트 통과해?" | GitHub CI | 객관적 사실 |
| "스펙대로 됐어?" | PR diff + 사람 | 최종 판단 |

**판별 기준**: *결정*이 필요하면 설계, *실행*만 남았으면 구현.

---

## 6. 실패 모드

| 증상 | 원인 | 대응 |
|---|---|---|
| 구현이 자꾸 다른 걸 만든다 | 스펙에 "하지 말 것"이 없다 | 금지 항목을 명시 |
| 구현이 범위를 넘어 리팩터링한다 | 수정 대상 파일이 안 적혀 있다 | 파일 목록을 닫힌 집합으로 |
| "다 됐습니다" 하는데 안 된다 | 완료 기준이 주관적 | 실행 가능한 명령으로 (`npm test`) |
| 설계자가 구현까지 해버린다 | 스펙 쓰기보다 고치는 게 빨라 보임 | 그 판단이 드는 순간이 곧 스펙 완성 시점 |
| 같은 버그가 반복된다 | 검증이 회귀 테스트를 안 남김 | 완료 기준에 회귀 테스트 필수화 |

## 7. 구조

```
agent-taskbook/
├── GUIDE.md                      # 이 문서 — 왜 이렇게 하는가
├── WORKFLOW.md                   # 단계별 상세 절차 — 어떻게 하는가
├── AGENTS.md                     # 구현자(Open Code) 규칙
├── prompts/
│   ├── _TEMPLATE-feature.md      # 기능 핸드오프 규격
│   ├── _TEMPLATE-bugfix.md       # 버그픽스 핸드오프 규격
│   └── YYYY-MM-DD-<slug>.md      # 실제 작업 스펙
└── .github/workflows/            # 검증 게이트
```

핸드오프 문서는 **버전 관리 대상**이다. PR에 스펙과 구현이 함께 올라가면,
리뷰어는 "이게 맞는 코드인가" 대신 "이게 스펙대로인가"를 볼 수 있다. 훨씬 쉬운 질문이다.
