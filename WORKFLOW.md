# 단계별 상세 절차

[GUIDE.md](GUIDE.md) 의 워크플로우를 실제로 도는 방법. 각 단계는 **산출물**과 **통과 조건**을 갖는다.
통과 조건을 못 채우면 다음 단계로 넘어가지 않는다.

```
0. 준비  →  1. 설계  →  2. 스펙 검토  →  3. 구현  →  4. 검증  →  5. 판정  →  6. 정리
            (Claude)     (사람)          (OpenCode)  (GitHub)   (사람)
```

---

## 0단계 — 준비 (최초 1회)

### 0.1 저장소 초기화

```powershell
git init
git branch -M main
```

### 0.2 구현자 측 규칙 고정 — `AGENTS.md`

Open Code는 저장소 루트의 `AGENTS.md` 를 자동으로 읽는다.
매번 프롬프트로 "범위 밖 작업 하지 마"라고 말하는 대신 **여기에 못 박는다.**

```markdown
# 구현 규칙

이 저장소에서 너는 **구현자**다. 설계는 이미 끝났다.

## 작업 방식
- 작업 지시는 `prompts/` 의 스펙 문서 하나로 온다. 그 문서가 유일한 요구사항 출처다.
- 문서의 "수정 대상 파일" 목록은 **닫힌 집합**이다. 그 밖의 파일은 건드리지 않는다.
- 문서의 "완료 기준"을 전부 만족시키면 끝이다. 그 이상은 하지 않는다.

## 멈추고 보고해야 하는 경우
- 스펙의 설명과 실제 코드가 다를 때
- 스펙이 모호해서 두 가지 이상으로 해석될 때
- 완료 기준을 만족시키려면 목록 밖 파일을 고쳐야 할 때

위 상황에서 **추측해서 진행하지 말고 멈춘다.** 되묻는 게 정답이다.

## 금지
- 범위 밖 리팩터링 / 포맷팅 변경
- 테스트를 통과시키기 위해 테스트를 수정하는 것
- 스펙에 없는 의존성 추가
```

### 0.3 검증 게이트 — `.github/workflows/verify.yml`

CI가 없으면 3단계는 "구현자의 자기 신고"로 전락한다. **가장 먼저 세운다.**

```yaml
name: verify
on: [push, pull_request]

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npm run typecheck
      - run: npm test
```

> **통과 조건**: 빈 커밋을 push 했을 때 CI가 초록불이다.
> 여기서 CI를 못 돌리면 이후 모든 단계의 검증이 무의미하다.

---

## 1단계 — 설계 (Claude Code)

### 1.1 탐색

스펙을 쓰기 전에 코드를 읽는다. **읽지 않고 쓴 스펙은 추측이다.**

```
로그인 실패 시 에러 메시지가 화면에 안 뜬다.
아직 고치지 말고, 관련 코드 흐름부터 파악해줘.
어디서 에러가 발생하고 어디서 삼켜지는지 경로:라인으로 알려줘.
```

### 1.2 스펙 작성

원인이 파악됐으면 문서를 만든다.

```
파악한 내용으로 prompts/2026-07-16-login-error-message.md 를 작성해줘.
prompts/_TEMPLATE-bugfix.md 형식을 따라줘.

이 문서만 읽고 구현할 수 있어야 해. 우리 대화 내용은 전달되지 않는다고 가정하고 써줘.
```

**산출물**: `prompts/YYYY-MM-DD-<slug>.md`

> **왜 날짜를 붙이나**: 스펙은 그 시점의 코드베이스에 대한 판단이다.
> 3개월 뒤 그 문서를 보는 사람이 "이게 지금도 유효한가"를 판단할 근거가 된다.

---

## 2단계 — 스펙 검토 (사람) ← **가장 중요한 단계**

여기서 5분 쓰면 뒤에서 한 시간을 아낀다. 반대로 여기를 건너뛰면
**틀린 스펙이 CI를 통과한 채로 머지된다.** 그게 최악의 결과다.

### 검토 체크리스트

| 항목 | 확인 |
|---|---|
| **자립성** | 이 프로젝트를 처음 보는 사람에게 이 문서만 줘도 구현이 되는가? |
| **파일 목록** | "수정 대상 파일"이 비어있거나 `path/to/...` 플레이스홀더로 남아있지 않은가? |
| **완료 기준** | 실행 가능한 명령인가? ("잘 동작한다" ❌ / `npm test` 통과 ✅) |
| **금지 항목** | "하지 말 것"이 채워져 있는가? 비어있으면 구현자는 뭐든 해도 된다고 읽는다. |
| **근본 원인** | (버그픽스) 증상을 가리는 게 아니라 원인을 고치는 스펙인가? |
| **경계** | 한 PR에 담길 크기인가? 넘치면 스펙을 쪼갠다. |

### 통과 못 하면

Claude Code로 **되돌아간다**. 구현에 넘기고 나서 고치는 게 아니다.

```
"수정 대상 파일에 아직 플레이스홀더가 있어. 실제 경로로 채워줘."
"완료 기준 3번이 주관적이야. 검증 명령으로 바꿔줘."
```

> **통과 조건**: 체크리스트 전 항목 ✅. 그리고 스펙을 커밋한다.
> ```powershell
> git add prompts/2026-07-16-login-error-message.md
> git commit -m "spec: 로그인 에러 메시지 미표시 수정"
> ```
> 구현 전에 스펙을 먼저 커밋해야 PR에서 "스펙 → 구현" 순서가 남는다.

## 3단계 — 구현 (Open Code)

### 3.1 브랜치

```powershell
git checkout -b fix/login-error-message
```

### 3.2 새 컨텍스트에서 실행

**반드시 새 세션.** Claude Code와 같은 창에서 이어서 하면 컨텍스트 분리가 깨지고,
이 방식의 이점이 통째로 사라진다.

```powershell
opencode
```

### 3.3 지시

```
prompts/2026-07-16-login-error-message.md 를 읽고 구현해줘.
AGENTS.md 의 규칙을 따라줘.
```

이게 전부다. 프롬프트가 길어진다면 **스펙이 부실하다는 신호**다 — 2단계로 돌아간다.

### 3.4 구현자가 멈췄을 때

"스펙과 코드가 다릅니다" / "이 부분이 모호합니다" 라고 멈추면 — **정상 동작이다.**
여기서 Open Code에게 답을 주지 말고 **1단계로 되돌아간다.**

```
❌ (Open Code에게) "아 그건 그냥 B로 해줘"
✅ (Claude Code에게) "구현 중에 이런 모순이 나왔어: [내용]. 스펙을 고쳐줘."
```

구두로 채워넣은 결정은 문서에 안 남는다. 다음 사람은 같은 자리에서 또 막힌다.

### 3.5 커밋 & PR

```powershell
git add -A
git commit -m "fix: 로그인 실패 시 에러 메시지 표시"
git push -u origin fix/login-error-message
gh pr create --fill
```

> **통과 조건**: PR이 열렸다. (로컬 테스트 결과는 통과 조건이 **아니다**. 판정은 CI가 한다.)

---

## 4단계 — 검증 (GitHub)

### 4.1 CI

```powershell
gh pr checks --watch
```

기다린다. **로컬에서 통과했다는 사실은 근거가 아니다.**
로컬은 구현자가 만진 환경이고, CI는 아무도 안 만진 환경이다.

### 4.2 diff 리뷰

CI 초록불이 켜졌으면 사람이 diff를 본다. **질문은 하나뿐이다:**

> "이 diff가 스펙대로인가?"

"이게 좋은 코드인가"가 아니다. 그건 1단계에서 이미 결정했다.

| 확인 | 방법 |
|---|---|
| 파일 목록 준수 | 스펙의 "수정 대상 파일" 밖 파일이 diff에 있는가? |
| 금지 항목 준수 | 범위 밖 리팩터링 / 포맷팅 노이즈가 섞였는가? |
| 완료 기준 충족 | 회귀 테스트가 실제로 그 버그를 잡는가? |

```powershell
gh pr diff
```

## 5단계 — 판정 (사람)

**어디로 되돌릴지가 이 방식의 핵심이다.** 잘못 되돌리면 같은 실수가 반복된다.

| 상황 | 되돌아갈 곳 | 이유 |
|---|---|---|
| CI 빨간불 | **3단계** (Open Code) | 구현이 스펙을 못 지켰다 |
| diff에 범위 밖 변경 | **3단계** (Open Code) | 규칙 위반 |
| CI 초록불, 근데 원하던 게 아님 | **1단계** (Claude Code) | **스펙이 틀렸다** |
| 스펙 전제가 코드 현실과 다름 | **1단계** (Claude Code) | 탐색이 틀렸다 |
| 전부 OK | **6단계** | |

### 되돌림 프롬프트

**3단계로** (구현 문제):
```
CI가 실패했어: [로그 붙여넣기]
스펙의 완료 기준을 다시 확인하고 고쳐줘. 스펙 범위는 그대로야.
```

**1단계로** (설계 문제):
```
구현은 스펙대로 됐고 CI도 통과했는데, 실제로는 [문제]가 남아있어.
스펙이 틀렸다는 뜻이야. prompts/2026-07-16-login-error-message.md 를 다시 봐줘.
```

> ⚠️ **가장 흔한 실수**: 설계 문제를 Open Code에게 되돌리는 것.
> 구현자를 붙잡고 "아니 그거 말고 이렇게" 를 반복하면 코드는 결국 고쳐지지만
> **스펙은 틀린 채로 머지된다.** 다음 작업은 그 틀린 문서를 근거로 시작된다.

---

## 6단계 — 정리

```powershell
gh pr merge --squash --delete-branch
```

스펙과 구현이 한 PR에 함께 머지된다. 이게 이 방식의 최종 산출물이다 —
**"이 코드가 왜 이렇게 됐는지"가 커밋 히스토리에 문서로 남는다.**

### 회고 (스펙이 틀렸던 경우에만)

5단계에서 1단계로 되돌아갔다면, 왜 스펙이 틀렸는지 한 줄 남긴다.
템플릿에 항목을 추가할 거리가 여기서 나온다.

```
"이번에 스펙이 틀린 이유가 뭐였지? _TEMPLATE-bugfix.md 에 보강할 항목이 있으면 추가해줘."
```

> 템플릿은 **과거에 틀린 방식들의 누적 기록**이다. 시간이 지날수록 스펙이 덜 틀리게 된다.

## 요약: 각 단계 통과 조건

| 단계 | 담당 | 산출물 | 통과 조건 |
|---|---|---|---|
| 0. 준비 | 사람 | `AGENTS.md`, CI | 빈 커밋에 CI 초록불 |
| 1. 설계 | Claude Code | `prompts/*.md` | 문서 존재 |
| 2. 검토 | 사람 | (커밋된 스펙) | 체크리스트 전 항목 ✅ |
| 3. 구현 | Open Code | PR | PR 열림 |
| 4. 검증 | GitHub | CI 결과 + diff | CI 초록불 |
| 5. 판정 | 사람 | 되돌림 or 승인 | 스펙대로임 |
| 6. 정리 | 사람 | 머지 커밋 | — |
