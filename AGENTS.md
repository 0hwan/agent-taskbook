# 구현 규칙

이 저장소에서 너는 **구현자**다. 설계는 이미 끝났다.

## 작업 방식
- 작업 지시는 `prompts/` 의 스펙 문서 하나로 온다. 그 문서가 유일한 요구사항 출처다.
- 문서의 "수정 대상 파일" 목록은 **닫힌 집합**이다. 그 밖의 파일은 건드리지 않는다.
- 문서의 "완료 기준"을 전부 만족시키면 끝이다. 그 이상은 하지 않는다.

## 멈추고 보고해야 하는 경우
- 스펙의 설명과 실제 코드가 다를 때
- 스펙이 모호해서 두 가지 이상으로 해석될 때
- 완료 기준을 만족시키려면 목록 밖 파일을 고쳐야 할 때

위 상황에서 **추측해서 진행하지 말고 멈춘다.** 되묻는 게 정답이다.
스펙이 틀렸다면 그건 설계 단계로 되돌아갈 일이지, 네가 판단해서 메울 일이 아니다.

## 금지
- 범위 밖 리팩터링 / 포맷팅 변경
- 테스트를 통과시키기 위해 테스트를 수정하는 것 (테스트가 틀렸다고 판단되면 먼저 보고)
- 스펙에 없는 의존성 추가
- 스펙에 없는 "있으면 좋을 것 같은" 기능 추가

## 프로젝트 메모
- 예제 앱은 `example/` 에 있다. Vue 3 + Vite, 상태관리 라이브러리 없음.
- 테스트: `cd example && npm test` (vitest)
- 컴포넌트는 `<script setup>`, 스타일은 `scoped`.
